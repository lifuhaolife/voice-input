# CPU 占用优化说明

## 问题分析

### 原有实现的问题

1. **频繁创建 WebSocket 连接**
   - 每次按下快捷键都创建新的 `XunfeiStreamer` 实例
   - 每次都要建立新的 WebSocket 连接（TCP 握手 + TLS 握手 + WebSocket 握手）
   - 每次都要创建新的线程

2. **资源浪费**
   - 连接建立耗时：约 0.5-1 秒
   - CPU 峰值：连接建立时 CPU 占用高
   - 内存碎片：频繁创建销毁对象

## 优化方案

### 核心思路：WebSocket 连接复用

不再每次录音都创建新连接，而是：
1. 程序启动时建立一次连接
2. 每次录音只重置状态并发送新的首帧
3. 程序退出时才关闭连接

### 代码修改

#### 1. `xunfei.py` - XunfeiStreamer 类

**新增方法：**
- `reset()` - 重置识别状态，清空队列
- `_send_start_frame()` - 发送首帧（用于复用连接）
- `cleanup()` - 清理资源（程序退出时调用）

**修改方法：**
- `start()` - 检测到已连接时，只重置状态并发送首帧
- `stop()` - 添加 `close_connection` 参数，默认保持连接

#### 2. `main.py` - StreamingVoiceInput 类

**修改逻辑：**
- `start()` - 初始化时创建识别器并建立连接
- `_on_hotkey_press()` - 只调用 `streamer.start()`（复用连接）
- `_on_hotkey_release()` - 调用 `streamer.stop(close_connection=False)`
- `stop()` - 调用 `streamer.cleanup()` 清理资源

## 性能提升

### 预期效果

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|
| 启动耗时 | ~0.5-1s | ~0.01s | 98% ↓ |
| CPU 峰值 | 高 | 低 | 显著降低 |
| 内存占用 | 波动大 | 稳定 | 更稳定 |

## 注意事项

1. **连接保活**
   - 讯飞 WebSocket 有超时机制（通常 60 秒无活动会断开）
   - 如果长时间不使用，连接可能被服务端关闭
   - 代码会自动检测并重新连接

2. **错误处理**
   - 如果连接断开，`start()` 会自动重新建立连接
   - 不影响用户使用体验

3. **兼容性**
   - 保持了原有的所有功能
   - API 接口完全兼容

## 进一步优化建议

如果 CPU 占用仍然较高，可以考虑：

1. **增加音频块大小**
   ```yaml
   recording:
     chunk_ms: 80  # 从 40ms 增加到 80ms
   ```

2. **增加批量发送数量**
   ```yaml
   xunfei:
     batch_chunks: 4  # 从 2 增加到 4
   ```

3. **使用更高效的音频处理**
   - 考虑使用 C 扩展进行音频转换
   - 减少 numpy 数组拷贝

## 回答你的疑问

> "使用 websocket 连接长连接、为什么每次都要 connect、close 呢？"

**原因：**
- 原代码设计问题，没有考虑连接复用
- 每次在 `_on_hotkey_press()` 中创建新的 `XunfeiStreamer` 实例
- 每次在 `_on_hotkey_release()` 中销毁实例并关闭连接

**现在的改进：**
- 只在程序启动时 connect 一次
- 每次录音只是重置状态并发送新的首帧
- 只在程序退出时 close 连接

这样就真正实现了"长连接"的优势！
